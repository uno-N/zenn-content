## 1. GitとGitHubの知識フェーズ

### 1-1. 「Git」と「GitHub」は別物

「Git」は「GitHub」の略称ではなかった。似た名前ですが、実は役割は全くの別物です。

### 1-2. コミット・プッシュとは

この言葉すこぶる使っていた。しかし、今、理解できました。

- **コミット**：Gitで作ったコードをローカルに記録するだけ
- **プッシュ**：GitHubにアップロードしてネット上に保管してくれる

### 1-3. それぞれの機能

**Gitとは：ローカルのバージョン管理システム**

- 「いつ」「何を」変更したかを記録
- この機能があることで過去のバージョンに戻せる

**GitHubとは：クラウドのコード共有プラットフォーム**

- コードをクラウドに保管
- 他の人と共有でき、複数人で作業できる仕組みがある
- Gitの内容もここから確認できる

## 3. GitとGitHubの実装フェーズ

GitHubにソースコードを保存する。

### 3-0. 2つのアプローチがあるが、どっち？

**① ローカル→GitHub（今回やる方）**

```
ローカルで開発 → コミット → GitHub作成 → push
```

- 既存のプロジェクトを後からGitHub管理にできる
- ネット環境がなくてもローカルで作業を進められる
- Gitの基本操作を順を追って学べる

**② GitHub→ローカル（もう一つの方法）**

```
GitHub作成 → git clone → 開発 → コミット → push
```

- 最初からリモートとの接続が完了している
- `git clone` すれば自動的に `.git` フォルダやremote設定が完了
- チーム開発を始めるときはこっちが主流

今回は①で進めていきます。

### 3-1. Gitでコミットする手順

```bash
# 3-1-1. 現在の状態を確認
git status

# 3-1-2. 変更をステージングエリアに追加
git add .

# 3-1-3. コミット（記録）する
git commit -m "Initial commit: React環境構築完了"
```

**3-1-1. 現在の状態を確認**

プロジェクトの階層にいるかを確認してください。

```bash
cd my-first-react-app
```

```bash
git status
```

**つまづきポイント①：ターミナルが占領されてコマンドが打てない**

> 原因：開発サーバーが起動しているターミナルではコマンド入力ができない
> 解決策：新しいターミナルを開く

`git status` を実行すると、赤色で `src/App.js` が表示されます。
赤色は、まだステージングされていない（コミットするファイルが選択されていない）という意味。

**3-1-2. ステージングエリアに追加**

```bash
git add .
```

**つまづきポイント②：`git add` と `git add .` の違い**

- `git add`：ファイル名を指定する必要がある
- `git add .`：「.（ドット）」= すべての変更を追加

「 . 」には「全て」という意味があります。入力を忘れないように注意。

**3-1-3. コミット（記録）する**

```bash
git commit -m "Initial commit: React環境構築完了"
```

**3-1-4. GUIでコミットする（参考）**

VSCode左側のソースコントロールアイコンからもコミット可能です。

完了！これでGitにプロジェクトが記録されました。

### 3-2. GitHubにプッシュする手順

コミットが完了したら、次はGitHubにアップロードします。それをプッシュという。

**3-2-1. GitHubで（空の）リポジトリを作成**

1. GitHubアカウントにログイン
2. ホーム画面右上の「+」→「New repository」
3. リポジトリ名を入力（例：`my-first-react-app`）
4. Public/Privateを選択（最初はPrivateでも良い）
5. **「Add a README file」は Off**（ローカルに既にファイルがある場合は空のリポジトリで作成）
6. 「Create repository」クリック

**3-2-2. ローカルとGitHubを接続**

```bash
# 1. リモートリポジトリを登録
git remote add origin https://github.com/ユーザー名/リポジトリ名.git

# 2. ブランチ名をmainに変更
git branch -M main

# 3. GitHubにプッシュ
git push -u origin main
```

成功すると以下のような表示が出ます：

```
Writing objects: 100% (26/26), done.
Total 26 (delta 2), reused 0 (delta 0)
To https://github.com/ユーザー名/リポジトリ名.git
 * [new branch]      main -> main
```

**3-2-3. GitHubのリポジトリページをリロード**

プロジェクトが表示されました。これでGitHubでバージョン管理ができます。

## 4. 【本題】ブランチ戦略の学習フェーズ

### 4-1. ブランチとは何か

突然出てきたワードですが、ブランチとはGitの機能です。
私は、こやつの扱いが不慣れだった故に多くの不具合生成機になってしまいました、、、。心して聞いてください。

#### 4-1-1. ブランチ = コードの並行世界

本番（main）を動かしながら、新しい機能追加や修正をすることができます。

```
main（本番）      ●――――――――●（本番として稼働中）
                  │
                  └─●―●―●  branch
                        │
                        ├─ dev-A病院/新機能A（Aさん担当）
                        ├─ dev-B病院/新機能B（Bさん担当）
                        └─ fix/バグ修正（Cさん担当）
                               ↓
                             テスト中
```

:::message alert
mainを直接編集するのはNGです。GitHubから直接編集するのもNGです。自分のローカルにCloneして接続してから編集します。
:::

### 4-2. 当時の開発作業フロー（おそらく）

```
dev-病院名-修正内容（私の作業ブランチ）
 │ ← プルリクエストを送る（私）
 │   → プルリクエストを承認（代表）
 │
 │ ← マージ（リードSEさん）
develop（テスト環境）
 │ ← マージ（リードSEさん）
main（本番環境）
```

**私の作業手順：**
1. GitHubでブランチ作成
2. VSCodeで同期、作成したブランチへ移動
3. テスト用ファイル修正
4. コミット・プッシュ
5. 出力画面確認（テスト環境URLで）
6. プルリクエスト送る（レビュアーを代表にして）

代表がもう一度出力画面を確認しプルリク承認。
リードSEさんがマージを実施。

すると、本番環境で不具合発生。といった感じでした。

この不具合を**コンフリクト（衝突）**というそうですが、その「コンフリクト」という言葉も知らなかったのです。

不具合を全て代表とリードSEさんで直したと聞いた時は肝が冷えました。余計に作業を増やすことに、、、、

### 4-3. コンフリクトとプルリクエスト、マージとは

- **コンフリクト**：同じコード部分が、違う内容でそれぞれ変更されていてどっちを選んでいいかわからない状態。古いコードを読み込んでしまうと起こる。
- **プルリクエスト（Pull Request）**：「この変更を取り込んでください」というお願い。
- **マージ（Merge）**：ブランチの変更を別のブランチ（mainなど）に統合すること。

コンフリクトしているとマージができない。

### 4-4. 結局、何が原因でコンフリクトが起きていたのか

最初、本番環境がおかしいと言われたので、私は「site.jsonの環境設定を間違えたのかも」と考えました。

**仮説①：本番環境のsite.jsonを編集してしまった？**
→ 違った。同じフォルダ内をコピペしていたから、環境設定は同じはず。テスト環境でちゃんと動いていたのが証拠。

**仮説②：間違ったブランチで作業していた？**
→ 違った。VSCodeで必ずブランチを確認していた。ブランチ名も修正ファイルも間違っていなかった。

**仮説③：他の病院のファイルをコピペした？**
→ 違った。同じ病院のフォルダ内で作業し、出力画面もテスト環境で正しく表示されていた。

**じゃあ、何が原因だったのか？**

記事を書いている際に、GitHubのブランチ作成画面を見て、もしかしたらと気づきました。

**仮説④：Sourceをmainにしていた気がする、、、（最終結論）**

「Source」という項目。ルートとなるmainをSourceにするのが良いと思っていましたが、**最新のブランチをSourceにすることで、コンフリクトが防げたのかもしれません。**

最新のコードとは、ちゃんと同期してコードを書き始めるだけでは足りなかったのです。

```
main（1週間前の古い状態）
  ↓ ここから切った
dev-ABC病院-修正
  ↓ この間にdevelopは最新になっている
develop（最新・他の人の変更が入っている）
  ↓ マージしようとする
コンフリクト！！
```

**つまり：**
1. mainから切った古いコードで作業開始
2. developには他の人の最新変更が入っている
3. 自分のブランチとdevelopで同じファイルの同じ場所が違う内容に
4. マージしようとしてコンフリクト大量発生

**なぜテスト環境で動いたのか？**
私のブランチ内では整合性が取れていたから。古いコードだけど、その中では矛盾がない状態でした。

### 4-5. Gitからやり直して学んだこと

**私に足りなかったのは：**
- ✗ コンフリクトが何か知らない
- ✗ どうやったら最新のコードから作業を始められるか知らない

これらを理解せずに作業すると：

| | 状態 |
|---|---|
| ✅ | 自分では正しくやったつもり |
| ✅ | テスト環境でも動いた |
| ❌ | でも古いコードから始めている |
| ❌ | マージ時にコンフリクト |
| ❌ | 本番環境に悪影響 |
| ❌ | チームに迷惑をかける |

個人開発でも、Gitの基礎は大切。

今回、ようやく「Gitからやり直した方がいい」の意味が理解できました。当時は悔しかったけど、今となっては貴重な学びの機会だったと思います。

## 5. まとめ

### 5-1. 今回の学び

**GitとGitHubの基礎**
- ✅ GitとGitHubは別物 - Gitはローカル、GitHubはクラウド
- ✅ コミット・プッシュの違い - コミットは記録、プッシュはアップロード
- ✅ ローカルからGitHubへの流れ - コード作成 → コミット → プッシュ

**ブランチ戦略**
- ✅ ブランチとは何か - コードの並行世界
- ✅ プルリクエストとマージ - お願いと統合の違い
- ✅ コンフリクトの仕組み - 同じ場所の違う変更による衝突

**謎の解明**
- ✅ 「Gitからやり直した方がいい」の真意
- ✅ なぜコンフリクトが起きていたのか
- ✅ Sourceの重要性 - mainではなくdevelopから切るべきだった

**暗記してサッと使いたいコマンド：**

```bash
git status                    # 現在の状態を確認
git add .                     # すべての変更をステージング
git commit -m "メッセージ"    # 変更を記録
git push origin ブランチ名    # GitHubにアップロード
git branch                    # ブランチ一覧を確認
```

### 5-2. つまずきポイントと解決法

**つまづきポイント①：ターミナルが占領されてコマンドが打てない**

- 解決：新しいターミナルを開く

**つまづきポイント②：`git add` と `git add .` の違い**
- `git add .` の「.（ドット）」を忘れずに

**つまづきポイント③：Sourceをmainにしていた（最重要）**
- 問題：古いmainから切っていた
- 結果：コンフリクト大量発生
- 解決：developの最新から切る ← **これが全ての原因だった**
