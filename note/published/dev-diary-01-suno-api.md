## 1. そもそもどうすれば既存のWebサービスを動かすことができるの？

まず、外から操作するために必要なものは、以下らしい。

- **APIエンドポイント**（どこにアクセスするか）
- **リクエスト形式**（どんなデータを送ると正しく処理されるのか）
- **認証情報**（アクセス権の証明）
- **実装手段**（どうやって送るか）

これらがあれば、Webサービスを動かすことができる。

### 1-1. APIエンドポイントを探せ！

そもそもAPI（Application Programming Interface）って何？ エンドポイントって何？

> **API (Application Programming Interface)**: アプリケーション同士が情報をやり取りするための「ルール」や「仕様」の集合体（お手紙の「送り方」のルール）。
> **エンドポイント**: そのルールに従って実際にやり取りが行われる「場所」や「入り口」（お手紙の「住所」）。URLのこと。

*by Gemini*

それは一体どこでわかるんだい、、

#### 1-1-1. 具体的な見つけ方

**方法1: 公式ドキュメントから（理想）**

サービスが公式APIを提供している場合（例：GitHub、Google Maps、OpenAI）

```
手順：
1. サービスの開発者向けページを探す
2. APIドキュメントを読む
3. エンドポイント一覧を確認
```

**方法2: Webページの開発者ツールから探す（リバースエンジニアリング）**

公式APIがない場合（Sunoのケース）

```
手順：
1. ブラウザで実際にサービスを使う
2. 開発者ツール（Network タブ）を開く
3. 実際の通信を観察
4. エンドポイントを発見
```

#### 1-1-2. 方法2：Webページから探す、で頑張る

SunoAI APIは公式の発表のものがなく、Webページから検証を開いて開発者ツールを使って探すことになりました。（全てはClaude大先生のご指示）

**開発者ツールを見てみる**

SunoAI使用中の実際のやり取りの中から、SunoAIがエンドポイントに送った指示を探すことができる。

1. SunoAIを開いたらそのままのページで開発者ページを開く
2. SunoAIで楽曲をCreateした後、NetworkタブにADDされたリクエスト（やりとりの履歴）を観察していく

探す内容は、

- Request MethodがPOST
- Request URL（これがエンドポイント）
- Payloadタブに曲を作る際に使用したプロンプト文が記載されている

**MethodがPOSTって何、、？**

- MethodはHTTPの通信方式のこと。（一旦深く考えない）
- POSTはサーバーへデータを送信すること。送ったデータで処理を行うことができる。

サーバーに何かリクエストを出すときにどんな内容の手紙なのかを明記して送信しなくてはいけないようだ。
訪問看護でいう「指示書依頼在中」みたいなこと（？）
（わかりずらい方が大半のはずなのにすいません、、。）

リクエスト一覧を見ているとGETというMethodも多くありました。
- GETはサーバーからデータを取得すること。Webページなどに結果を表示することができる。

**必要な情報が入ってるリクエストを発見！**

`v2-web/` という名前のリクエストが、探してたリクエストデータの様子。
プロンプト文が記載されており、このリクエストをもとにSunoAIは音楽生成を行ったようです。

このURL（エンドポイント）に指示を送ることで音楽生成を動かすことができそう。

```
APIエンドポイント：
https://studio-api.prod.suno.com/api/generate/v2-web/
```

**余談：エンドポイントの構造**

```
https://studio-api.prod.suno.com / api / generate / v2-web/
━━━━━━━━━━━━━━━━━━━━━━ ━━━ ━━━━━━━━ ━━━━━━
         ベースURL              パス    機能     バージョン
```

- **ベースURL**: サービスの住所
- **パス**: サービス内の部署
- **機能**: 何をするか（generate = 生成）
- **バージョン**: APIのバージョン（v2）

この構造を知っているからか、Claude大先生はずっと `/generate` を探せと言っていた。
「リクエスト名に `/generate` か `/music` か `/create` があるから、それを探したらある！！！」と何度も言われたが、フィルターするとうまく探せず、ずっとMethod POSTと書かれているリクエストの内容を、一つずつスクショして「これ？」と送り続けていた、、。

### 1-2. リクエスト形式（どんなデータを送ると正しく処理されるのか）を解明せよ！

#### 1-2-1. v2-web/ の Payload の中身を見てみる

**Payloadって何、、？**

> データ本体のこと: パケットやメッセージの中で、ヘッダー（宛先や制御情報）などを除いた、実際に送りたい情報（本文、ファイル、画像など）のこと。例：メール本文、Webサイトへのリクエストボディ、JSONデータなど。

```json
{
  "token": null,
  "generation_type": "TEXT",
  "title": "",
  "tags": "アコーディオンのメロディー, ダーティサウストラップ, 日本の女性ボーカル",
  "negative_tags": "",
  "mv": "chirp-auk-turbo",
  "prompt": "[Verse 1]\n\n陽だまりの中で笑う君と \nそよ風が運ぶ幸せのメロディ \n今日も一緒に歌おう、心の歌を \n優しい声が響くこの街で \n\n[Chorus]\n\nそれは温かい面白い気持ち \nまるで心が踊るように \n日常に輝きを添えて \nみんなの笑顔が広がる\n\n[Bridge]\n\n手を取り合って歩いて行こう \nどんな時も忘れないで \n君と共に描く物語 \n未来へと続くこの道を",
  "make_instrumental": false,
  "user_uploaded_images_b64": null,

    "web_client_pathname": "/create",

    "disable_volume_normalization": false,
    "can_control_sliders": ["weirdness_constraint", "style_weight"]
```

上記はPayloadの中身のJSON（データが整って入ってる型）です。View sourceを開くと見ることができる。

つまり、この形にデータを作って送れればOK！

#### 1-2-2. 本来の具体的なリクエスト形式の見つけ方

**方法1: 公式ドキュメント（理想）**

例：GitHub API → `https://docs.github.com/en/rest`

ドキュメントに以下が記載：
- 必要なパラメータ
- データ型
- サンプルコード

#### 1-2-3. 本アプリにおいての大事な発見

音響特性はそれぞれパラメータごとに並んでいると思っていたが、意外と `tags` というパラメータにまとめて置かれていた。

```json
// 予想
{
  "bpm": 120,
  "key": "C Major",
  "instrument": "Piano"
}

// 実際
{
  "tags": "Pop, 120 BPM, C Major, Piano"
}
```

→ **パラメータは全て `tags` という文字列にまとめる！**

### 1-3. 認証情報（アクセス権の証明）を取得せよ！

#### 1-3-1. 具体的な認証情報の見つけ方

認証情報は、会員カード。どこの誰でアクセスする権利があるかがわかる。これを必要なコード設定箇所に入れて使用する。

**方法1: Cookie（今回使用）**

ブラウザにログインすると自動的に発行される。

```
1. 開発者ツールのNetworkタブ → ページリロード
2. client?_clerk を含むリクエストを探す
3. Headers → Cookie の値をコピー
```

✅ メリット：ブラウザでログインするだけで取得可能、追加の手続き不要
❌ デメリット：有効期限が短い（数時間〜数日）、セキュリティリスクが高い、本番環境には不適切

**方法2: API Key（推奨）**

サービスが発行する専用のキー。

```
1. サービスにアカウント登録
2. 開発者ページでAPI Key発行
3. キーをコピー
```

✅ メリット：安定している（期限が長い）、管理しやすい、使用量の追跡が可能
❌ デメリット：サービスの公式サポートが必要、発行に手続きが必要な場合がある

今回は検証のみを行いたかったためCookieを使用してみました。

#### 1-3-2. 方法1：Cookieを取得しよう

1-1、1-2と同じように開発者ツールから探していきます。
同じくNetworkタブから、リロードを行い、clerkを含む名前を探せとのこと。Cookieの記載を発見！

この長い文字列を使用していきます。
