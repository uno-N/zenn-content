# Zenn 記事作成ガイド

## 読者像
- エンジニア・開発者
- 技術的な再現性・正確さを求めている
- コードや構造を重視する
- 同じ技術スタックを使っている人

## Zennで求められること
- **再現性**：読者が同じ手順で試せること
- **正確さ**：技術用語・コードに誤りがないこと
- **構造の明確さ**：何を学べるか最初にわかること
- **コードの丁寧さ**：コマンドやコードは必ずコードブロックで

## 📝 記事執筆前（企画段階）

### テーマ・目的

- [ ] この記事で伝えたい**技術的な学び**は明確か？
- [ ] **読者ペルソナ**（誰に向けて書くか）を決めたか？
    - □ プログラミング初心者
    - □ React学習者
    - □ Python・API連携を学びたい人
    - □ 個人開発者
- [ ] 読者がこの記事から得られる**価値**は何か？
    - □ 環境構築・手順の再現
    - □ つまづきポイントの解決
    - □ 技術の概念理解

### キーワード・topics設定

- [ ] `topics`（タグ）を1〜5個決めたか？（小文字英数字）
    - 例）react, python, api, nodejs, 個人開発
- [ ] Zennで同じトピックの記事を検索して、差別化できているか確認したか？

## ✍️ 執筆中

### frontmatter（必須）

```yaml
---
title: "タイトル"
emoji: "🎵"             # 記事のアイコン絵文字（1文字）
type: "tech"            # tech（技術）or idea（アイデア・考察）
topics: ["react"]       # 最大5つ、小文字英数字
published: false        # 公開時にtrue
---
```

- [ ] コマンドはすべて**コードブロック**に入れているか？

### Zenn独自記法

```
:::message
補足・メモ（青色ボックス）
:::

:::message alert
注意事項・警告（赤色ボックス）
:::

:::details 折りたたみタイトル
折りたたみ内容（長いコードや補足情報に）
:::
```

- [ ] 注意事項に `:::message alert` を使ったか？
- [ ] 補足情報に `:::message` や `:::details` を活用したか？

## 💡 Tips

### 筆が進まないとき
- 完璧なコードを目指さない（動くコードで公開して後から改善）
- 「自分がつまづいたこと」をそのまま書く
- まずコードブロックだけ並べて、説明を後から書く

### ネタに困ったとき
- 開発中に調べたことを記事にする
- エラーの解決方法をまとめる
- 「公式ドキュメントだけじゃわからなかったこと」を書く

### NoteとZennの使い分け
- **体験・感情・ストーリー** → Note
- **手順・コード・技術解説** → Zenn
- 同じ開発体験でも両方に書けるネタがある
